# UI 设计规范

## 1. 文档目的

本规范用于指导当前开源博客产品的前台站点与后台管理端 UI 统一升级，目标不是做一套通用后台皮肤，而是形成**适合开源博客产品、兼顾内容气质与开发效率、具备辨识度的设计语言**。

当前代码中，前后台已具备基础页面骨架，整体偏深色、玻璃拟态、青蓝强调，但仍停留在“可用壳层”阶段，缺少足够鲜明的视觉立场、页面节奏和组件一致性。本规范用于把现状推进到可执行的高质量界面方案。

---

## 2. 设计总方向

### 2.1 核心设计概念

**设计关键词：技术编辑部 / 深夜控制台 / 冷静但不冷淡**

不是做成常见 SaaS 模板，也不是做成极客炫技站。

这套博客产品更适合的方向是：

- 前台像一份可长期阅读的「数字技术杂志」
- 后台像一个克制、有秩序的「内容控制台」
- 两端都保持技术产品的理性气质，但通过细节层次、字体、信息密度和局部高光，建立自己的识别度

### 2.2 风格结论

- **前台**：深色编辑部风格，强调阅读沉浸、内容秩序、封面与标题张力
- **后台**：深色运营控制台风格，强调结构清晰、状态可读、操作高频效率
- **共通语言**：深色基底、冷色高光、清晰边框层级、卡片分层、强节奏标题、低饱和留白、谨慎动效

### 2.3 与当前现状的差异

当前前端风格存在以下问题：

1. 青蓝科技风方向是对的，但视觉层级过于单一
2. 卡片与页面外壳相似度过高，缺少“主次关系”
3. 前台首页仍像组件列表，不像真正的博客首页
4. 文章详情页偏信息堆叠，缺少阅读重心与辅助导航体系
5. 后台页面可用，但像默认后台模板，缺少内容产品特征

因此本次 UI 升级重点不是换颜色，而是**重建信息层级、页面气质和组件系统**。

## 3. 品牌与视觉气质

### 3.1 品牌印象

用户首次进入站点，应感受到以下印象：

- 这是面向开发者的内容产品
- 站点有自己的判断，不是套模板生成的博客皮肤
- 内容是主角，但界面并不乏味
- 前后台属于同一产品，不是两套割裂系统

### 3.2 识别锚点

建议建立 3 个长期识别锚点：

1. **高对比标题排版**：前台关键标题使用更具编辑感的展示字体
2. **冷色高光系统**：青蓝不是满屏铺开，而是集中用于关键状态、交互入口、焦点边界
3. **分层面板体系**：大面板、次面板、弱面板、虚线面板四层结构明确

## 4. 色彩系统

### 4.1 设计原则

- 主背景统一偏深，不走纯黑
- 强调色不泛滥，控制在 10%~15% 视觉面积内
- 文本优先保证长时间阅读舒适度
- 前台与后台同源，但前台更有情绪，后台更偏理性

### 4.2 推荐主色板

#### 中性色

- `bg-0`：#071018 —— 页面主背景
- `bg-1`：#0B1523 —— 一级容器背景
- `bg-2`：#101C2B —— 二级卡片背景
- `bg-3`：#162538 —— 悬浮/激活态背景
- `line-1`：rgba(255,255,255,0.08) —— 常规描边
- `line-2`：rgba(148,163,184,0.22) —— 强分隔线

#### 文本色

- `text-1`：#F3F7FB —— 一级标题
- `text-2`：#D6E0EA —— 正文/重点说明
- `text-3`：#94A6B8 —— 次级信息
- `text-4`：#64748B —— 弱提示

#### 品牌强调色

- `accent-primary`：#67E8F9 —— 主高光青
- `accent-primary-deep`：#22CFE6 —— 稳定操作色
- `accent-secondary`：#8B9CFF —— 次强调蓝紫，只用于局部结构点缀
- `accent-warm`：#F4B860 —— 少量暖色，用于阅读进度、数据提醒、封面辅助点

#### 功能状态色

- Success：#34D399
- Warning：#F59E0B
- Danger：#FB7185
- Info：#60A5FA

### 4.3 渐变规则

允许使用渐变，但只在以下区域使用：

- 首页 Hero 区背景氛围
- 文章头图区半透明光晕
- 后台关键统计卡的边缘高光
- 按钮 hover 的轻微内发光

禁止：

- 大面积高饱和紫蓝渐变铺满整页
- 所有卡片统一带渐变
- 所有按钮都发光

### 4.4 前后台色彩差异

- 前台：允许更多深青、墨蓝、少量暖灰，情绪更强
- 后台：降低装饰性，增加结构性灰阶，强调状态色和信息秩序

## 5. 字体系统

### 5.1 原则

避免继续使用过于普通的系统黑体组合作为全部层级字体。博客产品的辨识度，首先来自标题气质。

### 5.2 推荐组合

#### 方案建议

- **中文展示标题**：优先选择更有编辑感的中文标题字体，例如霞鹜文楷系以外更稳的现代黑体，或思源黑体加粗高字距策略
- **英文字母/数字/导航辅助**：JetBrains Mono / IBM Plex Mono / Geist Mono 中选一种用于元信息和后台数据字段
- **正文**：思源黑体 / HarmonyOS Sans / PingFang SC 作为可读性主文本

### 5.3 使用策略

- 首页主标题、栏目标题、详情页文章标题：展示型标题风格
- 导航、小标签、日期、slug、统计数值：等宽或准等宽字体增强技术感
- 正文与表单：保持高可读，无需花哨

### 5.4 字级建议

- Hero 主标题：48~64
- 页面一级标题：32~40
- 模块标题：22~28
- 卡片标题：18~22
- 正文：15~17
- 辅助信息：12~14

## 6. 布局系统

### 6.1 栅格原则

- 前台采用**宽容器 + 内容主栏 + 辅助侧栏**的编辑型布局
- 后台采用**稳定侧边导航 + 顶部操作带 + 主内容栅格**
- 不追求满屏塞满，保留明显呼吸区

### 6.2 前台布局建议

- 页面最大宽度：1280~1360
- 文章详情正文列宽控制：720~780
- 首页内容区不建议所有模块等宽平铺，应建立主次节奏
- 搜索、分类、标签页应尽量统一结果列表骨架，减少视觉切换成本

### 6.3 后台布局建议

- 左侧导航固定宽度：240~264
- 顶栏高度明确，保持全局操作一致
- 列表页遵循：过滤区 → 批量摘要区 → 表格/卡片主体 → 分页
- 设置页采用双栏或分组折叠，避免超长单列表单

### 6.4 留白原则

- 大区块之间使用 32/40/48 间距
- 卡片内部常规使用 20/24/28 间距
- 页面顶部首屏区域比下方内容更宽松

## 7. 圆角、边框、阴影

### 7.1 圆角体系

- 页面大容器：24~28
- 常规卡片：18~20
- 表单控件：12~14
- 标签/胶囊按钮：9999

### 7.2 边框体系

- 所有主要容器都应有轻边界，不要完全依赖阴影
- 强交互区域可增加内描边或 hover 外发光
- 虚线边框专门用于占位、空状态、辅助说明区

### 7.3 阴影体系

- 深色场景阴影应偏“环境雾感”，而不是传统硬阴影
- 主卡片可使用大范围低透明阴影
- 后台表格行 hover 用背景变化优先，不依赖阴影

---

## 8. 组件风格规范

### 8.1 按钮

按钮分四类：

1. 主按钮：高亮青色填充，服务于创建、保存、搜索、确认
2. 次按钮：深底描边，服务于次级操作
3. 危险按钮：低饱和红边框 + hover 增亮
4. 纯文本按钮：用于页内轻操作、切换、辅助导航

规范：

- 主按钮避免过矮，建议 40~44 高度
- hover 不是简单变亮，应有轻微上浮或边界增强
- 同一视图最多只保留 1 个主按钮焦点

### 8.2 输入框

- 深色背景下输入框边界必须清晰
- focus 不仅变色，还应有外圈光晕或内描边
- 长表单场景使用 label 在上、输入框在下，不用占位符代替标签

### 8.3 卡片

卡片是本产品的核心视觉组件，至少分四类：

1. **Hero 卡**：首页主叙事、强视觉
2. **内容卡**：文章列表、统计、设置分组
3. **数据卡**：后台统计卡、状态板
4. **说明卡**：空态、提示、系统说明

### 8.4 标签/状态胶囊

- 分类标签偏结构色
- 标签云偏低饱和描边
- 后台状态标签必须颜色与语义稳定绑定
- 不同状态标签尺寸保持一致，避免列表跳动

### 8.5 表格

后台表格不应只是标准管理台表格。

规范：

- 表头可更轻，但分隔更清晰
- 行 hover 要明显
- 重要字段左重右轻
- 操作按钮尽量收敛成 2~3 个高频动作
- 支持在移动端退化为卡片列表

### 8.6 空状态

空状态不能只写“暂无数据”。

建议包含：

- 当前没有什么
- 为什么可能出现
- 下一步可做什么

## 9. 动效原则

### 9.1 总原则

动效服务于节奏与层级，不做炫技。

### 9.2 推荐动效

- 页面首屏模块分段淡入，上移 8~16px
- 卡片 hover 轻微抬升 + 边框增强
- 侧栏切换高亮平滑移动
- 搜索结果与评论状态切换使用短时 opacity/transform 过渡
- 成功反馈可用轻微描边闪现，不要弹跳

### 9.3 动效时长建议

- 微交互：120~180ms
- 面板出现：220~280ms
- 页面段落进入：300~420ms

### 9.4 禁止项

- 全站大面积持续浮动动效
- 背景粒子不停运动
- 列表页每次加载都过度动画
- 后台高频操作场景使用夸张动画

## 10. 前后台共通设计语言

### 10.1 共通点

前后台必须共享以下语言：

- 相同深色底层世界观
- 相同强调色逻辑
- 相同边框、圆角、阴影体系
- 相同按钮与输入框基本规则
- 相同状态标签语义
- 相同信息优先级：标题 > 主信息 > 辅助信息 > 说明信息

### 10.2 差异点

#### 前台

- 更强调内容展示与阅读沉浸
- 更允许视觉叙事与版式张力
- 更注重封面、标题、节奏、目录、推荐关系

#### 后台

- 更强调效率、扫描性、状态明确
- 视觉更克制，少用大面积氛围效果
- 重视操作反馈、筛选区、列表结构、异常提示

### 10.3 统一建议

前后台不建议完全两套 token。建议一套基础 token + 两套场景变量：

- Foundation Token：颜色、间距、字号、圆角、阴影、动效时长
- Public Semantic Token：首页、文章、阅读辅助、标签云、评论区
- Admin Semantic Token：统计、表格、表单、审核、设置分组

## 11. 页面级改版方向

### 11.1 首页

当前问题：

- 更像“文章接口聚合页”，不是产品首页
- 缺少明确主叙事区
- 分类、标签、文章层级平

改版方向：

- 顶部加入更有记忆点的 Hero 区
- 首页首屏建立“站点定位 + 精选文章 + 搜索入口”三段式结构
- 最新文章不要单纯垂直列表，可做“1 个主推 + 多个次级文章卡”
- 标签入口可做热度层级，避免纯按钮堆砌
- 加入作者理念/站点特色模块，增加开源博客的人味

### 11.2 文章详情页

当前问题：

- 阅读区、辅助区、评论区之间关系较平
- 缺目录、阅读进度、上下文推荐等强辅助结构

改版方向：

- 顶部文章头图区改为“标题 + 摘要 + 元信息 + 标签”编辑式头部
- 正文列宽更克制，提升长文可读性
- 右侧增加目录、阅读信息、站内继续浏览入口
- 评论区视觉从表单堆砌转为“讨论模块”
- 增加上一篇/下一篇或相关文章模块

### 11.3 分类 / 标签 / 搜索页

当前问题：

- 页面之间差异小，缺少各自语义
- 搜索缺少结果概览和筛选氛围

改版方向：

- 分类页突出“主题归档”语义
- 标签页突出“话题网络”语义，可使用不同权重标签样式
- 搜索页突出“检索工作台”语义，搜索框、结果统计、关键词高亮应更清晰
- 三类列表页面使用统一结果卡结构，但头部模块各自区分

### 11.4 后台仪表盘

当前问题：

- 统计卡和最近内容块可用，但还像默认后台首页
- 缺少真正有行动导向的信息组织

改版方向：

- 仪表盘上半区改为“运营概览 + 待处理事项 + 快捷操作”
- 统计卡强化状态层级，不只显示数字
- 最近文章与最近评论呈现更像工作台，不是普通列表
- 可加入“今日状态”“站点健康”“内容节奏”等轻运营视图

### 11.5 后台文章管理页

当前问题：

- 筛选区与表格区关系传统，缺少重点信息标识
- 表格信息密度高但缺少视觉锚点

改版方向：

- 顶部加入页面摘要：文章总数、草稿数、已发布、最近更新
- 筛选区做成工作台条带
- 列表中标题作为主视觉，状态与分类清晰贴边
- 操作列收敛，强化“编辑”“预览”“发布状态”三件事
- 移动端用卡片列表降级

### 11.6 后台评论审核页

当前问题：

- 已有左右结构，但审核动作层次还不够强
- 状态切换与评论详情关系可继续优化

改版方向：

- 顶部先给审核池概览和风险提示
- 左侧列表像“待办池”，右侧详情像“审核面板”
- 审核动作做成更明确的决策区
- 敏感信息、命中规则、父评论、上下文文章应分块展示
- 对待审核评论突出视觉优先级

### 11.7 后台设置页

当前问题：

- 当前为大表单堆叠，缺少信息架构
- 配置预览和编辑关系偏弱

改版方向：

- 改为分组导航 + 表单主体 + 右侧说明/实时预览
- 按“站点 / SEO / 评论 / 内容策略”建立明确分区
- 每组增加说明、风险提示、影响范围
- 开关类配置统一成设置行，不与文本字段混排

## 12. 设计 Token 建议

以下建议后续沉淀为 `frontend/src/design/tokens.ts` 或 CSS Variables。

### 12.1 Foundation Token

- 颜色：`--bg-0 --bg-1 --bg-2 --line-1 --text-1 --accent-primary ...`
- 字号：`--font-size-12/14/16/18/20/24/32/40/56`

#### 前台

- `--public-hero-bg`
- `--public-card-bg`
- `--public-article-width`
- `--public-tag-active`
- `--public-reading-accent`

#### 后台

- `--admin-shell-bg`
- `--admin-panel-bg`
- `--admin-stat-highlight`
- `--admin-warning-bg`
- `--admin-table-hover`

## 14. 实施原则

### 14.1 不一次性大改

本轮先出规范与任务单，不直接大规模改代码。后续实施建议按页面优先级拆分 commit。

### 14.2 先建立基础，再改页面

先做：

1. 设计 token
2. 布局壳层升级
3. 通用组件升级
4. 页面改版落地

### 14.3 保持工程可维护

- 避免把大量样式写死在页面里
- 尽量沉淀为公共容器、标题、标签、状态组件
- 前后台共用基础 token，不复制两份主题系统

---

## 15. 结论

这套 UI 方向的核心不是“更炫”，而是**更像一个真正的开源博客产品**：

- 前台要有内容产品的叙事感与阅读气质
- 后台要有内容控制台的秩序与效率感
- 两端统一在一套深色、冷静、技术编辑部风格的设计语言下

后续前端实现，应优先围绕首页、文章详情页、后台仪表盘、后台文章管理页建立新的视觉骨架，再逐步扩展到搜索、评论审核、设置等页面。
