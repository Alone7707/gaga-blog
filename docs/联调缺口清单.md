# 联调缺口清单

## 本次检查范围

- 文档：`docs/` 下现有需求拆解、进度、UI 改版相关文档
- 前端：公开站点页面、后台仪表盘 / 文章 / 评论 / 设置主链路
- 后端：公开文章 / 分类 / 标签 / 搜索 / 评论 / 归档接口，以及后台文章 / 评论 / 设置 / 仪表盘接口

## 已完成的小修

### 1. 修复分类 / 标签总览页非法 slug 半空态

- **问题**：`CategoriesPage`、`TagsPage` 仅在 query 为空时回退首项，query 带非法 slug 时，高亮卡片区不会展示。
- **影响**：页面不会直接崩溃，但会出现“列表区正常、入口区空白”的半空态，清爽版 UI 下更明显。
- **修复**：前端统一增加 `normalizedActiveSlug`，当 query 缺失或非法时，自动回退到首个有效分类 / 标签。

### 2. 补上分类 / 标签文章页侧栏失败提示

- **问题**：`CategoryPostsPage`、`TagPostsPage` 的侧栏入口请求失败时仅结束 loading，没有任何提示。
- **影响**：主内容区可能正常，但左侧导航静默空白，联调时难以判断是无数据还是接口失败。
- **修复**：补充 `sidebarErrorMessage`，在侧栏区域明确展示“分类入口加载失败 / 标签入口加载失败”。

### 3. 补强后台评论邮箱空值兜底

- **问题**：后台评论页类型定义和 `maskEmail` 默认假设 `authorEmail` 一定是字符串。
- **影响**：若后端返回 `null`（例如匿名评论或后续契约放宽），评论列表与详情区都可能出现运行时异常。
- **修复**：前端类型改为 `string | null`，并在邮箱脱敏函数中增加空值兜底，统一显示 `--`。

### 4. 修复公开文章详情接口包裹层不一致

- **问题**：后端 `GET /api/public/posts/:slug` 返回结构为 `{ post: ... }`，但前端 `getPublicPostDetail` 直接按详情对象消费。
- **影响**：文章详情页 `PostDetailPage` 直接拿到包裹对象时，`title`、`summary`、`tags`、`contentMarkdown` 等字段会变成 `undefined`，页面容易整块空白或局部渲染异常。
- **修复**：前端 API 层统一解包，改为 `.then((response) => response.post)`。

### 5. 补强公开评论列表的空值保护

- **问题**：`PostDetailPage` 直接把 `response.list` 赋值给 `comments`。
- **影响**：若接口未来短暂返回异常结构、代理层丢字段，评论区 `reduce / length / replies` 相关渲染存在崩溃风险。
- **修复**：增加 `Array.isArray(response.list) ? response.list : []` 兜底。

## 主要联调发现

### 一、前后端字段 / 包裹结构

#### 1. 公开文章详情曾存在包裹层不一致

- **状态**：已修
- **位置**：
  - 后端：`backend/src/modules/public-post/public-post.controller.ts`
  - 前端：`frontend/src/api/public.ts`

#### 2. 后台接口整体包裹风格不完全统一，需要 UI 改版时继续收口

目前存在两类返回方式：

- 直接返回对象：如 `GET /api/admin/posts`、`GET /api/admin/settings`、`GET /api/admin/dashboard/overview`
- 带业务键包裹：如 `GET /api/admin/posts/:id -> { post }`、`GET /api/admin/comments/:id -> { comment }`、`GET /api/admin/comments/stats -> { stats }`

**风险**：

- UI 改版阶段如果新页面直接按“统一接口风格”猜测，很容易再次出现解包错误。
- 建议后续补一份 `docs/API返回约定.md`，明确“列表接口 / 详情接口 / 动作接口”的统一 envelope 规范。

### 二、前端易崩空值点

#### 1. 后台评论邮箱字段对 null 已做前端兜底，但后后端契约仍建议明确

- **位置**：
  - 前端：`frontend/src/pages/admin/AdminCommentsPage.vue`
  - 类型：`frontend/src/types/comment.ts`
  - 后端：`backend/src/modules/comment/comment.service.ts`
- **状态**：前端已修
- **现状**：前端已将 `authorEmail` 调整为 `string | null`，并在脱敏函数里统一对空值显示 `--`；但后端当前仍直接透传数据库值，接口契约文档未明确“必为字符串”还是“允许 null”。
- **风险**：后续如果评论导入、匿名评论或历史脏数据进入系统，不同页面可能继续对该字段作不同假设。
- **建议**：在接口契约中明确后台评论邮箱字段是否允许为 `null`，避免后续页面再次各自兜底。

#### 2. 设置页未覆盖公开侧读取联调

- **位置**：
  - 前端：`frontend/src/pages/admin/AdminSettingsPage.vue`
  - 后端：`backend/src/modules/setting/public-site.controller.ts`
  - 路由 / 前台：`frontend/src/router/index.ts`、`frontend/src/layouts/PublicLayout.vue`
- **现状**：后台设置页已接入管理端读写，但前台尚未接入 `GET /api/public/site`、`GET /api/public/site/overview`、`GET /api/public/site/pages/about`；导航里已有 `/about` 入口，路由却未注册对应页面。
- **影响**：
  - 设置保存后，首页标题、副标题、页脚、关于页内容均不会实时反映；
  - 点击前台“关于”会直接落入兜底重定向回首页，存在明显断链；
  - UI 改版阶段如果继续围绕首页与 About 做视觉升级，容易在后期才暴露消费链路未接通。
- **建议优先级：高**
  - 尽快补 `site.ts` / `site.ts 类型` 与 About 页面；
  - 首页头图、页脚文案优先切到公开站点配置；
  - 至少完成“后台设置保存 -> 前台 About / 首页可见”闭环验证。

### 三、关键链路缺口

#### 1. 文章编辑页只走“创建 / 更新”，未接入独立发布动作

- **位置**：
  - 前端：`frontend/src/pages/admin/AdminPostEditorPage.vue`
  - 后端：`backend/src/modules/post/post.controller.ts`
- **现状**：后端已有：
  - `PATCH /api/admin/posts/:id/publish`
  - `PATCH /api/admin/posts/:id/unpublish`
  - `PATCH /api/admin/posts/:id/archive`
- **但前端**：仍通过保存表单时直接改 `status` 字段完成状态切换。
- **影响**：
  - UI 改版如果要加“发布 / 下线 / 归档”快捷按钮，当前前端链路还没真正走动作接口。
  - 后端 `publish` 接口里有单独的校验与发布时间处理，前端现在没有覆盖到这条链路。
- **建议优先级：高**
  - UI 改版前应明确：状态切换是“保存表单”还是“独立动作”。
  - 更推荐切换到独立动作接口，避免编辑保存和发布动作语义混用。

#### 3. 评论审核页只有审核动作，没有后台回复能力

- **现状**：
  - 前台支持提交评论和回复评论；
  - 后台支持审核评论、看回复链；
  - 但后台没有“管理员直接回复评论”的能力。
- **影响**：若后续产品希望管理员在后台直接运营互动，目前链路是断的。
- **建议优先级：中**
  - 若当前版本只做审核，可保留；
  - 若 UI 改版中准备突出评论运营，则需要新增接口与页面交互。

#### 4. 归档页接口已完成，但与设置项 `content.archivePageSize` 尚未打通

- **位置**：
  - 后端：`backend/src/modules/public-archive/public-archive.service.ts`
  - 设置：`content.archivePageSize`
- **现状**：归档接口当前一次性返回全部公开文章并按年月分桶；设置里的归档分页大小暂未参与公开归档接口。
- **影响**：
  - 数据量上来后，前端首屏渲染成本会增加；
  - 设置项目前更偏“预留字段”，没有真正落到公开链路。
- **建议优先级：中**
  - 短期可继续保留；
  - 若 UI 改版要重做归档页，建议一并决定是否需要分页 / 折叠加载。

### 四、工程与环境缺口

#### 1. 当前环境无法直接跑前端构建校验

- **现状**：本地执行 `npm run build` 失败，原因是 `npm` 不在 PATH。
- **影响**：本轮只能做静态代码联调检查，无法在当前环境完成最终编译验证。
- **建议**：主会话或后续环境 agent 补一次 Node / npm PATH 检查，然后至少执行：
  - `frontend: npm run build`
  - `backend: npm run build`

## 建议的 UI 改版前联调顺序

1. **先统一 API 返回约定文档**：避免新 UI 再踩解包错误。
2. **补前台设置消费闭环**：确认首页 / About / 页脚真实字段来源。
3. **明确文章状态流转方案**：是继续“保存时改状态”，还是切换到独立 publish/unpublish/archive 按钮。
4. **补侧边栏失败提示与非法 slug 回退**：这些是低成本高收益的稳定性增强。
5. **环境补 build 验证**：至少保证 UI 改版前基线可编译。

## 本轮结论

- 当前仓库的主链路已经具备“可联调”的最小形态。
- 最大的真实问题是**接口返回包裹层不一致导致的公开文章详情页消费错误**，已修复。
- 接下来更值得做的不是大改 UI，而是先把**接口约定、状态动作链路、设置到前台的闭环**补齐。
