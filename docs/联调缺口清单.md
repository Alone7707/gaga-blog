#### 2. 公开文章详情包裹层不一致问题已收口

- **状态**：已修并已文档化
- **位置**：
  - 后端：`backend/src/modules/public-post/public-post.controller.ts`
  - 前端：`frontend/src/api/public.ts`
- **结论**：前端已统一按 `{ post }` 解包。

#### 3. 后台文章发布动作契约已对齐

目前后台文章状态流转已明确分为两类请求：

- 内容保存：`POST /api/admin/posts`、`PATCH /api/admin/posts/:id`
- 独立动作：`PATCH /api/admin/posts/:id/publish|unpublish|archive`

**结论**：

- 编辑页与列表页都已切到独立动作接口。
- 编辑保存不再混入 `status`，避免和后端发布动作语义冲突。
- 发布前先保存内容，再调用发布动作，符合后端现有契约。

#### 4. 后台接口风格仍不完全统一，但已可被文档明确约束

目前仍存在两类返回方式：

- 直接返回对象：如 `GET /api/admin/posts`、`GET /api/admin/settings`、`GET /api/admin/dashboard/overview`
- 带业务键包裹：如 `GET /api/admin/posts/:id -> { post }`、`GET /api/admin/comments/:id -> { comment }`、`GET /api/admin/comments/stats -> { stats }`

**结论**：

- 该现状当前不构成功能阻塞；
- 但新页面联调时仍不能按“全部统一风格”想当然接入；
- 本轮已通过文档把该风险点显式化。

### 三、仍然存在的关键缺口

#### 1. 评论审核页只有审核动作，没有后台回复能力

- **现状**：
  - 前台支持提交评论和回复评论；
  - 后台支持审核评论、看回复链；
  - 但后台没有“管理员直接回复评论”的能力。
- **影响**：若后续产品希望管理员在后台直接运营互动，目前链路仍断。
- **建议优先级：中**
  - 若当前版本只做审核，可保留；
  - 若要做评论运营，需新增接口与页面交互。

#### 2. 归档页接口已完成，但与设置项 `content.archivePageSize` 尚未打通

- **位置**：
  - 后端：`backend/src/modules/public-archive/public-archive.service.ts`
  - 设置：`content.archivePageSize`
- **现状**：归档接口当前一次性返回全部公开文章并按年月分桶；设置里的归档分页大小暂未参与公开归档接口。
- **影响**：
  - 数据量上来后，前端首屏渲染成本会增加；
  - 设置项目前更偏“预留字段”，没有真正落到公开链路。
- **建议优先级：中**
  - 若后续继续做体验精修，建议决定是否分页 / 折叠加载。

#### 3. 后台 API 层 normalize 仍未完全覆盖

- **现状**：
  - 公开接口已在 `frontend/src/api/public.ts`、`frontend/src/api/site.ts` 做了一轮 normalize；
  - 后台 API 层目前只有文章列表做了显式 `list/pagination` 兜底。
- **影响**：
  - 当后台接口未来加字段或短暂返回异常结构时，仪表盘 / 评论 / 设置页仍更依赖页面层防御。
- **建议优先级：中**
  - 可在下一轮收口中补 dashboard/comments/settings/taxonomy 的最小 normalize。

#### 4. 前端个别类型与后端空值事实仍有轻微偏差

- **现状示例**：
  - `frontend/src/types/dashboard.ts` 中 `recentComments.authorEmail` 当前定义为 `string`
  - 但后端 `DashboardService` 实际返回可为 `null`
- **影响**：
  - 现在未必立刻出错，但这是典型的“文档 / 类型 / 代码事实”后续可能再漂移的点。
- **建议优先级：低到中**
  - 后续做类型清扫时可一并收口。
